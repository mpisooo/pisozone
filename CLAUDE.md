# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Cos'è

PisoZone: PWA in italiano per il tracciamento dell'attività fisica, con social (amici, messaggi, gruppi, feed), gamification (crediti, 10 livelli, 18 medaglie, sfide giornaliere, streak con "freeze") e notifiche push. Tutta l'UI e i messaggi di commit sono in italiano.

## Comandi

```bash
npm run dev                 # dev server Vite (http://localhost:5173)
npm run build               # tsc + build di produzione
npm test                    # Vitest (vitest.config.ts dedicata; un file: npx vitest run src/lib/challenges.test.ts)
npm run typecheck           # type-check del client (copre SOLO src/)
npm run typecheck:api       # type-check delle funzioni /api in NodeNext (intercetta import senza .js)
```

CI in `.github/workflows/ci.yml`: typecheck ×2 + test + build a ogni push/PR — è il semaforo da controllare prima del deploy (che resta manuale). Niente lint configurato. `gh` CLI non installata: stato delle run via `curl https://api.github.com/repos/mpisooo/pisozone/actions/runs?per_page=1`.

## Deploy — ATTENZIONE

Il progetto Vercel `pisozone-app` **NON è collegato a GitHub**: il push su main non deploya nulla. Deploy manuale:

```bash
npx vercel --prod --yes     # CLI già autenticata; produzione = https://pisozone-app.vercel.app
```

Le migrazioni DB (`supabase-schema/supabase-schema-vN.sql`, incrementali, numerate) **le esegue l'utente a mano nel SQL editor di Supabase** — Claude non ha accesso SQL diretto. Ogni modifica allo schema = nuovo file vN+1, mai modificare i file già eseguiti. Scrivere codice client tollerante all'ordine deploy/migrazione (es. campi opzionali nel tipo finché la colonna non esiste).

## Architettura

**Client React 19 + Vite + Tailwind v4 (no config file) che parla direttamente a Supabase via RLS.** Non c'è un backend proprio: la logica server vive in (a) trigger e RPC Postgres, (b) poche funzioni serverless Vercel in `api/` con service role.

- **Auth**: username+password su Supabase Auth con email finta `{username}@pisozone.local`. L'email di recupero, se verificata, SOSTITUISCE quella finta in `auth.users.email`.
- **Stato condiviso**: `context/ProfileContext.tsx` è l'unica fonte del profilo (`hooks/useProfile.ts` è solo un re-export). Il suo `updateProfile` fa **upsert dell'oggetto completo** con default espliciti: aggiungere un campo a `Profile` richiede attenzione (campo opzionale se la colonna può non esistere ancora).
- **Streak**: UNICA implementazione in `lib/challenges.ts` → `calcStreak(activities, protectedDates)` dove protectedDates = freeze da `useStreakFreeze` + giorni di riposo da `useRecovery` (v33): tutti e tre i chiamanti (Home, Calendar, Challenges) passano `[...frozenDates, ...restDates]`. Non ricalcolarlo a mano (regressione già successa in Calendar.tsx).
- **Sfide giornaliere**: `hooks/useDailyChallenges.ts` è l'unica fonte (Home e Challenges la condividono); i template in `lib/challenges.ts` (`CHALLENGE_POOL`, selezione deterministica per utente+data via PRNG). Le sfide devono essere completabili in giornata (mai basate su streak multi-giorno).
- **Crediti/livelli/temi/cornici**: mutazioni SOLO via RPC atomiche Postgres (`unlock_next_level`, `purchase_theme`, ...), mai update diretti dei crediti.
- **Push**: insert su `messages`/`friendships` → webhook Supabase (header `x-webhook-secret`) → `api/webhooks/*` → web-push. Bloccare l'insert (rate limit) blocca anche la notifica. Il promemoria serale (`api/cron/reminder.ts`, cron Vercel alle 20:00 UTC) e i due webhook rispettano le preferenze granulari dell'utente (v28): 3 toggle per categoria (`notif_reminder_enabled`/`notif_messages_enabled`/`notif_friend_requests_enabled`) + una fascia oraria di silenzio comune (`notif_quiet_start`/`notif_quiet_end`, fuso Europe/Rome, avvolge la mezzanotte). Logica pura e testata in `api/_lib/notificationPrefs.ts` (`allowsNotification`/`isQuietHour`) + `api/_lib/time.ts` (`getRomeHour`/`getRomeTodayRange`, condivisi da cron e webhook); l'invio effettivo passa da `sendPushToUserIfAllowed` in `api/_lib/push.ts`. UI in `NotificationSettingsCard.tsx`.
- **Rate limiting**: trigger Postgres `enforce_insert_rate_limit()` (v23+v24) su messages/group_messages/friendships, conteggio sul registro insert-only `rate_limit_events`. L'errore ha prefisso `RATE_LIMIT`, riconosciuto da `lib/errors.ts` → i hook mostrano toast dedicati. Sulle `/api` c'è anche un limiter per IP best-effort (`api/_lib/rateLimit.ts`).
- **GDPR**: pagine pubbliche `/privacy` e `/termini` (fuori da ProtectedRoute); export dati client-side (`lib/dataExport.ts`); cancellazione account in `api/account/delete.ts` (identità dal JWT, mai dal body); `profiles.terms_accepted_at` NULL = ConsentGate bloccante.
- **Tracciamento GPS** (v29, roadmap punto 12): "a schermo acceso" per corsa/bici/camminata (`GPS_TRACKABLE_TYPES`) — il background reale non è fattibile su iOS PWA standalone (WebKit sospende la pagina a schermo bloccato), quindi serve tenere lo schermo attivo. `useGpsTracking` orchestra `watchPosition` + Wake Lock, con tempo trascorso calcolato sempre da timestamp reali (mai da un contatore) così una sospensione imprevista non lo sfasa. Logica pura testata in `lib/gps.ts` (`haversineMeters`, `isPlausibleSample` per scartare i rimbalzi GPS, `computeRouteStats`, `projectToViewBox` per la sagoma SVG del percorso — niente mappe/tile esterni, solo la forma). Calorie da velocità media (non MET fisso) via `calcCaloriesFromSpeed` in `lib/constants.ts`. UI in `WorkoutTrackingOverlay.tsx`: schermata bloccata di default, uno slide-to-unlock nativo (`<input type="range">`, accessibile "gratis") rivela Pausa/Termina per pochi secondi — evita che i tocchi accidentali col telefono in tasca fermino il tracciamento. Percorso salvato punto-per-punto in `activity_routes` (owner-only, cascata da `activities`), mostrato in sola lettura in `ActivityEditModal` se `activity.gps_tracked`.
- **Log palestra strutturato** (v32, roadmap v2 pilastro 02 punto 1): tabella `exercise_sets` a cascata da `activities` — una riga per "blocco" di lavoro (esercizio + serie × ripetizioni allo stesso carico; `weight_kg` NULL = corpo libero, non un peso ignoto), RLS owner-only CON policy di DELETE perché la modifica è delete+reinsert (`replaceActivityExercises`), a differenza dei punti GPS immutabili. Logica pura testata in `lib/exerciseSets.ts`: le bozze dell'editor sono stringhe grezze (ogni campo può restare vuoto, stesso principio delle metriche percepite) convertite con `draftsToEntries`/`draftStatus` (riga senza nome = scartata in silenzio, riga con nome ma serie/rip mancanti = segnalata come incompleta); PR con `buildPrMap`/`detectNewPrs` (chiave = nome normalizzato minuscolo, PR = carico OLTRE il massimo storico, il pareggio no; annunciato SOLO a storico caricato — senza confronto ogni carico sembrerebbe un record); `buildGymRecords` e `exerciseSuggestions` per la card in Statistiche e la `<datalist>`. Persistenza in `lib/activityExercises.ts` (best effort DOPO l'insert dell'attività, come foto e percorso: fallimento = toast dedicato, attività comunque salva — tollera la migrazione non eseguita). `hooks/useExerciseHistory.ts` carica lo storico (2 colonne) per PR/suggerimenti/record; `appendLocal` dopo il salvataggio evita di ri-annunciare lo stesso PR nella stessa sessione. UI condivisa `components/ExerciseSetsFields.tsx` (controllata come PerceivedMetricsFields, `idPrefix` per id unici), mostrata solo per `palestra`; in ActivityEditModal il salvataggio esercizi SALTA se il fetch iniziale è fallito (`setsLoaded`), altrimenti il delete+reinsert cancellerebbe set mai visti. PR nel toast di salvataggio (riga ambra + `haptic('celebrate')`); card "Record palestra" in Stats (all-time, non segue il filtro periodo); `exercise_sets` nell'export GDPR con tolleranza tabella-assente (`optionalTables` in `dataExport.ts`).
- **Piani e programmi di allenamento** (v34, roadmap v2 pilastro 02 punto 4): i programmi sono TEMPLATE nel codice (`PLAN_CATALOG` in `lib/plans.ts`, stesso pattern di CHALLENGE_POOL/MEDALS, stringhe di dominio italiane incluse); il DB registra solo l'iscrizione (`plan_enrollments`: plan_key, started_on, completed_at/abandoned_at/credits_earned — UNA attiva per utente via indice parziale). L'avanzamento NON è persistito: `computePlanProgress` (puro, testato) lo deriva dalle attività — settimane a blocchi di 7 giorni da started_on, una sessione si sblocca nella SUA settimana (niente front-load), il recupero delle arretrate è ammesso, ogni attività salda al massimo una sessione. Crediti al completamento con lo stesso modello fiduciario delle sfide (trigger `award_plan_credits` security definer su UPDATE null→valorizzato di completed_at, importo bounded 0..300, non ripetibile né annullabile); `started_on` ancorato a oggi ±1 nella policy INSERT (partire nel passato permetterebbe di completare con attività già registrate). Client: `hooks/useTrainingPlan.ts` (start/abandon/claim, tollerante pre-migrazione), pagina `/plans` (`pages/Plans.tsx`: programma attivo con settimane e sessioni + catalogo), widget in Home sezione "Il tuo percorso" (sempre visibile: è l'unico entry point, niente voce in Navbar). `plan_enrollments` nell'export GDPR (optionalTables).
- **Recupero: riposo, idratazione, sonno** (v33, roadmap v2 pilastro 02 punto 5): tabella `recovery_logs`, una riga per utente per giorno (`unique(user_id, day)`, upsert con `onConflict: 'user_id,day'`) — `rest` boolean, `water_ml` e `sleep_hours` nullable per sempre (stesso principio di rpe/mood). Il giorno di riposo protegge la streak COME un freeze (entra in `calcStreak` con le frozenDates) ma è gratuito e si dichiara SOLO per il giorno corrente — retrodatarlo renderebbe inutili i freeze a 300 crediti (l'economia va preservata); limite 2 a settimana lunedì-domenica applicato dal client (`canMarkRest` in `lib/recovery.ts`, puro e testato insieme ai clamp acqua/sonno). `hooks/useRecovery.ts`: fetch 180 giorni, upsert ottimistico con rollback, espone `restDates`. Calendar mostra un pallino viola sui giorni di riposo + badge nel pannello giorno; l'offerta freeze in Home NON compare se ieri era riposo. Il cron del promemoria serale salta chi ha `rest=true` oggi (`getRomeToday` in `api/_lib/time.ts`, tollerante pre-migrazione). `recovery_logs` nell'export GDPR (optionalTables). Le medaglie restano su sole attività (i freeze già non contavano — coerente).

Le pagine importano il proprio namespace direttamente (`import log from '../lib/i18n/log'`), non da un barrel — l'accesso diretto a proprietà dà autocompletamento e controllo a compile-time. `src/lib/i18n/index.ts` assembla tutti i namespace in un unico oggetto `it` (tipo `Strings`): è il punto in cui in futuro si sceglierebbe il dizionario giusto in base a una preferenza utente/browser, oggi esiste solo l'italiano. Le stringhe interpolate sono funzioni (`credits: (n: number) => ...`), mai template literal nel componente. Gli import di `date-fns/locale` (`it`) restano invariati: sono formattazione date, un problema distinto dalle stringhe UI. `src/lib/i18n/it.test.ts` verifica che ogni foglia del dizionario sia una stringa o una funzione (mai `undefined`). **Regola permanente per tutto il codice scritto da qui in avanti**: ogni pagina o componente nuovo, o ogni stringa aggiunta a una pagina esistente, va scritta direttamente nel namespace i18n corrispondente (nuovo namespace se è una pagina nuova, aggiunta a quello esistente se è una funzionalità dentro una pagina già coperta) — mai testo italiano hardcoded nel JSX, nemmeno "temporaneamente". Se un namespace esistente non si adatta bene, crearne uno nuovo seguendo lo stesso pattern (oggetto tipizzato, `export default`, import diretto) e aggiungerlo a `index.ts`.

## Convenzioni in `api/`

- Import relativi **con estensione `.js`** (`from '../_lib/sentry.js'`) — ESM su Vercel, dimenticarla causa ERR_MODULE_NOT_FOUND in produzione (già successo).
- Ogni handler è avvolto in `withSentry` (`api/_lib/sentry.ts`); errori non-fatali via `captureError`.
- `api/_lib/push.ts` esporta `supabaseAdmin` ma al load inizializza web-push (richiede le VAPID env): nelle funzioni che non inviano push, creare un client admin locale invece di importarlo.
- Env server: `SUPABASE_SERVICE_ROLE_KEY`, `SUPABASE_WEBHOOK_SECRET`, `VAPID_*`, `SENTRY_*`; l'URL Supabase si legge da `VITE_SUPABASE_URL` (condivisa col client).

## Convenzioni UI

- Design system in `src/index.css`: classi `.card`, `.btn-primary`, `.input-dark`, `.skeleton`, heatmap `heatmap-0..4`. Font: Bebas Neue per i titoli (`.font-bebas`), Inter per il body. Gli stili inline `style={{...}}` sono la convenzione del codebase (ignorare i warning del linter).
- Materiali e profondità: `.card-hero` (mesh + grana, vedi Sistema Zone sopra) e `--glow-accent` (glow direzionale permanente su `.btn-primary`/`.btn-save`, non solo su `:hover` — su mobile l'hover non esiste) vanno usati con parsimonia sui momenti di punta, non come default.
- Movimento e aptica (roadmap v2, pilastro 01 punto 4): feedback tattile SOLO via `haptic('light'|'success'|'celebrate'|'error')` da `lib/haptics.ts`, MAI `navigator.vibrate` diretto — su iOS (il dispositivo dell'utente) non esiste e il modulo ripiega sul tick nativo dell'`<input type="checkbox" switch>` di Safari 17.4+. Token di movimento in `index.css` (`--dur-fast/base/slow`, `--ease-out`, `--ease-spring`): i modali entrano con `.overlay-fade` sul backdrop + `.modal-pop` (dialoghi centrati) o `.sheet-up` (bottom sheet); le schermate full-screen restano su `.page-enter`. I numeri importanti contano con `components/AnimatedNumber.tsx` (rAF, niente dipendenze). Navigazione con `viewTransition` (react-router) su Navbar e card di Home, chrome fermo via `view-transition-name` su `.topbar`/`.navbar` — progressive enhancement, dove non supportata resta `.page-enter`. `prefers-reduced-motion` è rispettato globalmente in coda a `index.css` (azzera anche le transizioni inline e le view transitions; AnimatedNumber fa lo stesso check via matchMedia): le animazioni nuove non devono aggiungere guardie proprie.
- Stati di caricamento SOLO con `SkeletonCard` / `SkeletonRow` (entrambi in `components/SkeletonCard.tsx`) — mai `animate-pulse` con div ad hoc (ripulito col punto 15).
- Ogni modale/popup DEVE seguire il pattern accessibile: `useFocusTrap(ref, open, onClose)` + `role="dialog"` + `aria-modal` + chiusura con Esc (riferimento: `ActivityEditModal.tsx`). Gli overlay auto-dismissivi usano invece `role="status"` + `aria-live`.
- Overlay su mobile (l'utente usa la PWA standalone su iPhone, `viewport-fit=cover`): elementi interattivi vicino ai bordi SEMPRE dentro la safe area (`env(safe-area-inset-top/bottom)` — la X di un lightbox a `top-4` finisce sotto orologio/batteria). Form lunghi = schermata fissa a tutto schermo con header sticky (mai bottom sheet che scorre tutto insieme). Ogni overlay con scroller interno DEVE bloccare il body (`document.body.style.overflow='hidden'` in useEffect con ripristino) e usare `overscroll-contain` + `min-h-0` sullo scroller, altrimenti su iOS lo scroll chaining aggancia la pagina dietro e il gesto "si blocca" a metà. Riferimento per tutti e tre: `ActivityEditModal.tsx`.
- Rotte in `App.tsx`: pubbliche (`/auth`, `/privacy`, `/termini`) fuori da ProtectedRoute; il resto dentro AppLayout con lazy loading per pagina.
- Errori utente: messaggi in italiano, concreti, con invito a riprovare (mai `error.message` grezzo).

## Gotcha noti

- Le env var inlined nel `define` di `vite.config.ts` hanno `.trim()` apposta: una newline finale nella key su Vercel rompeva SOLO il WebSocket realtime (sintomo: `%0A` nell'URL in console).
- `tsconfig.json` ha `include: ["src"]` — errori in `api/` non emergono dal type-check standard.
- Il service worker (`public/sw.js`) è cache-first: dopo un deploy l'HTML vecchio può sopravvivere finché la PWA non viene riaperta da zero.
- RLS su `profiles` permette la SELECT a tutti gli utenti autenticati (serve per la ricerca amici): non mettere dati sensibili in quella tabella senza rivedere le policy.
