# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Cos'è

PisoZone: PWA per il tracciamento dell'attività fisica — 15 sport, GPS, log palestra con record personali, programmi multi-settimana, obiettivi liberi, recupero (riposo/acqua/sonno) — con social (amici, messaggi, gruppi, feed con reazioni), gamification (crediti, 10 livelli, 22 medaglie, sfide giornaliere, streak con "freeze") e notifiche push. UI disponibile in italiano (default) e inglese (switch in Profilo, v3 pilastro 04); i messaggi di commit restano sempre in italiano.

## Comandi

```bash
npm run dev                 # dev server Vite (http://localhost:5173)
npm run build               # tsc + build di produzione
npm test                    # Vitest (vitest.config.ts dedicata; un file: npx vitest run src/lib/challenges.test.ts)
npm run typecheck           # type-check del client (copre SOLO src/)
npm run typecheck:api       # type-check delle funzioni /api in NodeNext (intercetta import senza .js)
npm run build:analyze       # build + stats.html (treemap dei chunk, mai committato/servito)
```

CI in `.github/workflows/ci.yml`: typecheck ×2 + test + build a ogni push/PR — è il semaforo da controllare prima del deploy (che resta manuale). Niente lint configurato. `gh` CLI non installata: stato delle run via `curl https://api.github.com/repos/mpisooo/pisozone/actions/runs?per_page=1`.

## Deploy — ATTENZIONE

Il progetto Vercel `pisozone-app` **NON è collegato a GitHub**: il push su main non deploya nulla. Deploy manuale:

```bash
npx vercel --prod --yes     # CLI già autenticata; produzione = https://pisozone-app.vercel.app
```

Le migrazioni DB (`supabase-schema/supabase-schema-vN.sql`, incrementali, numerate) **le esegue l'utente a mano nel SQL editor di Supabase** — Claude non ha accesso SQL diretto. Ogni modifica allo schema = nuovo file vN+1 (guardare la cartella per l'ultimo numero), mai modificare i file già eseguiti. Scrivere codice client tollerante all'ordine deploy/migrazione (es. campi opzionali nel tipo finché la colonna non esiste; fetch fallito su tabella nuova = dati vuoti, mai crash).

## Architettura

**Client React 19 + Vite + Tailwind v4 (no config file) che parla direttamente a Supabase via RLS.** Non c'è un backend proprio: la logica server vive in (a) trigger e RPC Postgres, (b) poche funzioni serverless Vercel in `api/` con service role.

### Fondamenta

- **Auth**: username+password su Supabase Auth con email finta `{username}@pisozone.local`. L'email di recupero, se verificata, SOSTITUISCE quella finta in `auth.users.email`.
- **Stato condiviso**: `context/ProfileContext.tsx` è l'unica fonte del profilo (`hooks/useProfile.ts` è solo un re-export). Il suo `updateProfile` fa **upsert dell'oggetto completo** con default espliciti: aggiungere un campo a `Profile` richiede attenzione (campo opzionale se la colonna può non esistere ancora; non includerlo nell'update finché `profile.<campo> === undefined`, o l'upsert fallirebbe pre-migrazione).
- **Logica in moduli puri**: i calcoli vivono in `src/lib/` testati con Vitest (`challenges`, `stats`, `zones`, `gps`, `exerciseSets`, `plans`, `goals`, `recovery`, `reactions`, ...), le pagine restano sottili. Le funzioni con dipendenza dal tempo prendono `now` come parametro per test deterministici. `lib/stats.ts` genera anche il CSV per Excel italiano (`;` e virgola decimale); `lib/dataExport.ts` ha i download JSON/CSV (BOM UTF-8 per Excel) e l'export GDPR completo (le tabelle post-v31 stanno in `optionalTables`: se mancano pre-migrazione l'export non fallisce; `activity_routes` è PAGINATA a blocchi da 1000 — PostgREST tronca silenziosamente oltre quel tetto e i percorsi GPS sono l'unica tabella che lo sfonda).
- **Streak**: UNICA implementazione in `lib/challenges.ts` → `calcStreak(activities, protectedDates)` dove protectedDates = freeze da `useStreakFreeze` + giorni di riposo da `useRecovery` (v33): tutti e tre i chiamanti (Home, Calendar, Challenges) passano `[...frozenDates, ...restDates]`. Non ricalcolarlo a mano (regressione già successa in Calendar.tsx).
- **Coda offline** (roadmap v2, pilastro 05, "offline-first robusto"): `addActivity` in `hooks/useActivities.ts` mette in coda (localStorage) un'attività se `!navigator.onLine` o se l'insert fallisce per un vero errore di rete (`postgrest-js` risponde con `status: 0` SOLO in quel caso — mai per un vero codice HTTP: vedi `isNetworkFailure` in `lib/offlineQueue.ts`). NIENTE Background Sync nel service worker: su iOS Safari (il dispositivo dell'utente) l'API non esiste — stesso limite già noto per il GPS "a schermo acceso". Si riprova quando l'app è davvero aperta (evento `online` + `visibilitychange`), non in background reale. Le attività in coda appaiono subito in UI come attività "pending" (`mergeWithPending`, id sintetico `pending-<uuid>`, crediti a zero finché non sincronizzate) — TUTTE le pagine che usano `useActivities()` (Home, Calendar, Stats, Challenges, Plans, Medals) le vedono già incluse nei loro calcoli, comportamento voluto (ottimistico). Sono modificabili ed eliminabili dal Calendario come le altre (roadmap v3, pilastro 04, "offline senza asterischi"): `updateActivity` in `useActivities` rileva l'id sintetico e aggiorna il payload dentro la coda invece di chiamare Supabase; il badge "In attesa di rete" resta, ma il tocco non è più disabilitato. Foto ed esercizi allegati mentre offline non vengono più scartati: `lib/offlineAttachments.ts` li tiene in IndexedDB (un `Blob` non è serializzabile nel JSON di `offlineQueue`/localStorage), indicizzati per `localId`; `flushQueue` li applica best effort appena l'attività ottiene un id vero (stesso pattern "dopo l'insert" di foto/esercizi/percorsi online), poi ripulisce la voce IndexedDB — anche se l'attività viene scartata per un errore vero, per non lasciare allegati orfani.
- **Sfide giornaliere**: `hooks/useDailyChallenges.ts` è l'unica fonte (Home e Challenges la condividono); i template in `lib/challenges.ts` (`CHALLENGE_POOL`, selezione deterministica per utente+data via PRNG). Le sfide devono essere completabili in giornata (mai basate su streak multi-giorno).
- **Crediti/livelli/temi/cornici**: mutazioni SOLO via RPC atomiche Postgres (`unlock_next_level`, `purchase_theme`, `use_streak_freeze`, ...) o trigger `security definer` (`award_challenge_credits`, `award_plan_credits`), mai update diretti dei crediti dal client.
- **UI one-shot via flag su `profiles`**: `push_prompt_seen` (prompt push), `terms_accepted_at` NULL → ConsentGate, `onboarding_seen` false → OnboardingTour, `news_seen_version` (v35) < `NEWS_VERSION` → `WhatsNewOverlay` (annuncio novità per utenti esistenti: VERSIONATO — per una nuova ondata si incrementa la costante in `WhatsNewOverlay.tsx` e si riscrivono gli item in `shell.whatsNew`; il tour al completamento allinea anche news_seen_version, ma SOLO se la colonna esiste). Stesso pattern per flag futuri: colonna con default, campo opzionale nel tipo TS finché la migrazione non è eseguita.
- **i18n + switch lingua IT/EN** (v3, pilastro 04, chiuso il 20/07/2026): tutte le stringhe UI vivono in `src/lib/i18n/` — un file per namespace (`common`, `shell`, `auth`, `home`, `log`, `calendar`, `stats`, `challenges`, `medals`, `profile`, `social`, `legal`, `recovery`, `plans`, `guide`, `goals`, `insights`, `wrapped`, `share`, `duels`, `seasonalEvents`, `notifications`, `recap` — 23 in totale), niente funzione `t('a.b.c')` a stringa. Ogni file ha DUE oggetti, `it` (`as const`) ed `en` (`Widen<typeof it>` — vedi sotto), esportati come `createNamespaceProxy(it, en)`: le pagine continuano a importare il namespace direttamente e staticamente (`import log from '../lib/i18n/log'`, non da un barrel), il Proxy sceglie l'oggetto giusto ad ogni lettura in base a `getLanguage()` — nessuna pagina è mai stata convertita a un hook `useStrings()`. `src/lib/i18n/language.ts` tiene lo stato (`localStorage` chiave `pz-language`, letture avvolte in try/catch perché Vitest gira in ambiente `node` senza `localStorage` e diversi moduli `lib/` puri — `dataExport`, `stats`, `shareCard`, `insights`, `perceivedMetrics` — importano namespace i18n) + `useLanguage()` (`useSyncExternalStore`). Il cambio lingua in tempo reale (switcher in Profilo → `LanguageSettingsCard.tsx`) non aggiorna le pagine già montate da solo: `AppLayout` in `App.tsx` chiama `useLanguage()` e mette `key={language}` sul `<div className="app-layout">`, forzando il remount dell'intero sottoalbero visibile (TopBar/Routes/Navbar) a ogni switch — badge/notifiche restano fuori dalla key, niente riconnessione realtime. **`Widen<T>` in `proxy.ts`**: `it` è `as const`, quindi ogni sua stringa è un tipo literal (`'Salva'`, non `string`) — `en: typeof it` non compila (`'Save'` non è assegnabile a `'Salva'`); `en` va sempre annotato `Widen<typeof it>` (allarga solo le foglie stringa e i tipi di ritorno delle funzioni a `string`, preservando struttura/nesting/arità — le chiavi mancanti o in eccesso restano un errore di compilazione). `it.test.ts` verifica sia che ogni foglia (in entrambe le lingue) sia stringa/funzione non vuota, sia che `en` rispecchi esattamente la forma di `it` namespace per namespace. Gli import di `date-fns/locale` restano invariati (formattazione date ≠ stringhe UI: le date sono in italiano anche con l'app in inglese, limite noto e accettato) e i numeri usano `toLocaleString('it-IT'|'en-US', ...)` a seconda della lingua dove serve un separatore decimale corretto. **Regola permanente**: ogni stringa nuova va nel namespace corrispondente (nuovo namespace per pagine nuove), SEMPRE in entrambe le lingue — mai testo italiano hardcoded nel JSX, nemmeno "temporaneamente". Eccezione voluta e mai cambiata: i DATI di dominio (titoli di sfide/medaglie/programmi/eventi stagionali in `CHALLENGE_POOL`/`MEDALS`/`PLAN_CATALOG`/`seasonalCatalog.ts`) restano hardcoded in italiano nei loro moduli `lib/`, fuori dal sistema i18n — anche con l'app in inglese quei nomi non si traducono.

### Allenamento

- **Foto attività** (v27): colonna `activities.photo_url` + bucket pubblico `activity-photos`, un file per attività al path stabile `{userId}/{activityId}.jpg` (sostituzione = upsert). Compressione client in `lib/activityPhotos.ts` (JPEG max 1600px); l'upload avviene DOPO l'insert dell'attività (serve l'id) e un suo fallimento NON annulla il salvataggio (toast dedicato) — è il pattern "best effort dopo l'insert" riusato anche da percorsi GPS ed esercizi. Il file non cade in cascata: rimozione manuale in `useActivities.deleteActivity` e `api/account/delete.ts`.
- **Tracciamento GPS** (v29): "a schermo acceso" per corsa/bici/camminata (`GPS_TRACKABLE_TYPES`) — il background reale non è fattibile su iOS PWA standalone (WebKit sospende la pagina a schermo bloccato). `useGpsTracking` orchestra `watchPosition` + Wake Lock, con tempo trascorso calcolato sempre da timestamp reali (mai da un contatore). Logica pura in `lib/gps.ts` (`haversineMeters`, `isPlausibleSample` per scartare i rimbalzi, `computeRouteStats`, `projectToViewBox` per la sagoma SVG). Calorie da velocità media via `calcCaloriesFromSpeed` (`lib/constants.ts`), non MET fisso. UI in `WorkoutTrackingOverlay.tsx`: schermata bloccata di default, slide-to-unlock nativo (`<input type="range">`) rivela Pausa/Termina per pochi secondi. Percorso punto-per-punto in `activity_routes` (owner-only, cascata da `activities`), sola lettura in `ActivityEditModal` se `activity.gps_tracked`. **Split per km** (roadmap v2, GPS potenziato): `computeSplits` in `lib/gps.ts` (blocchi da 1 km con istante di attraversamento interpolato sulla distanza; ultimo tratto < 1 km marcato `partial` con la sua distanza reale, sotto i 50 m scartato; campioni con timestamp non crescente ignorati) — `fetchActivityRoute` riporta anche `recorded_at` come `t` proprio per questo, quindi gli split sono retroattivi su ogni percorso v29+. UI sotto la sagoma in ActivityEditModal (barre comparative sul passo, split più veloce = barra piena), visibile solo con almeno 1 km completo. **Altimetria** (v42): colonna `altitude_m` nullable su `activity_routes` da `coords.altitude` (quota grezza salvata, filtraggio SOLO in lettura); `computeElevationProfile` in `lib/gps.ts` (media mobile + isteresi 3 m contro il rumore GPS, `null` se quota assente o sotto il 50% di copertura → sezione nascosta), grafico SVG puro in `ElevationProfileChart.tsx`. NON retroattiva (i percorsi pre-v42 non hanno la quota). `saveActivityRoute`/`fetchActivityRoute` sono tolleranti alla colonna assente pre-migrazione (retry senza `altitude_m`): mai perdere il percorso per colpa della quota. **Dislivello persistito + medaglie della montagna** (v44, roadmap v3): colonna `elevation_gain_m` su `activities` (interi, nullable per sempre — attività manuali o senza quota non ne hanno uno), calcolata al salvataggio in WorkoutTrackingOverlay con lo stesso `computeElevationProfile`; le 4 medaglie della montagna (Esploratore/Cartografo su `totalGpsKm`, Scalatore/Ottomila su `totalElevationGainM` in `achievementStats`) leggono colonna e flag `gps_tracked` invece di riscaricare i percorsi. Insert tollerante pre-migrazione: `insertActivity` in `useActivities` riprova senza la colonna (pattern `altitude_m`). **Dislivello manuale** (roadmap v3, pilastro 04, nessuna nuova migrazione — stessa colonna, fonte diversa): per gli sport in `ELEVATION_CAPABLE_TYPES` (`lib/constants.ts`: corsa, bici, camminata, trekking, arrampicata, motocross) Log.tsx e ActivityEditModal.tsx mostrano un campo `elevation_gain_m` a mano, utile quando non si traccia col GPS (arrampicata non è mai GPS-trackable). In ActivityEditModal il campo è **sola lettura** (nascosto, il valore salvato resta quello) se `activity.gps_tracked`: lì il D+ arriva dall'altimetria del percorso, non va sovrascritto. Le medaglie della montagna sommano il campo qualunque sia la fonte (GPS o manuale), `achievementStats` è già agnostico. **Percorso nel feed** (v45, roadmap v3): flag `route_visible` su activities (default FALSE, consenso esplicito per attività — mai retroattivo) + policy SELECT "friends read shared routes" su `activity_routes` (amici accettati non bloccati, vincolata al flag; quella owner-only resta in OR). Toggle nel recap dopo-allenamento (update ottimistico) e in ActivityEditModal (col salvataggio, chiave solo a colonna presente); nel feed `useFeed` carica best-effort le sagome delle attività condivise (max 10, solo lat/lng) e la card mostra `RouteShape` — MAI la mappa reale. **Mappa reale** (2026-07-16, chiude "GPS potenziato"): `RouteMap.tsx` = Leaflet + tile raster MapTiler (`streets-v2`/`-dark` secondo `html.light`), PRIMA dipendenza esterna della sezione GPS (decisione presa con l'utente: OSM pubblico scartato per policy). Caricata SOLO via `React.lazy` da ActivityEditModal (Leaflet ~150kB, chunk `vendor-leaflet` in `manualChunks`); senza `VITE_MAPTILER_KEY` o offline si ripiega sulla sagoma `RouteShape` (che resta la fallback permanente, non rimuoverla). La key è nel `define` di `vite.config.ts` (pattern `.trim()`), in `.env.local` e nelle env Vercel; è pubblicabile (come la anon key Supabase) ma va ristretta al dominio nella dashboard MapTiler. `public/sw.js` NON mette in cache `api.maptiler.com` (tile opache, gonfierebbero la quota storage). Il container della mappa ha `isolation: isolate` (i pane Leaflet arrivano a z-index 700 e coprirebbero l'header sticky del modale).
- **Log palestra strutturato** (v32): tabella `exercise_sets` a cascata da `activities` — una riga per blocco (esercizio + serie × ripetizioni allo stesso carico; `weight_kg` NULL = corpo libero), RLS owner-only CON policy DELETE perché la modifica è delete+reinsert (`replaceActivityExercises`). Logica pura in `lib/exerciseSets.ts`: bozze a stringhe grezze (ogni campo può restare vuoto) ⇄ righe valide via `draftsToEntries`/`draftStatus` (riga senza nome = scartata, riga incompleta = segnalata); PR con `buildPrMap`/`detectNewPrs` (chiave = nome normalizzato minuscolo; PR = carico OLTRE il massimo storico; annunciato SOLO a storico caricato); `buildGymRecords`/`exerciseSuggestions` per la card Statistiche e la `<datalist>`. Persistenza best effort in `lib/activityExercises.ts`; storico via `hooks/useExerciseHistory.ts` (`appendLocal` evita PR duplicati nella stessa sessione). UI condivisa `ExerciseSetsFields` (solo `palestra`); in ActivityEditModal il salvataggio esercizi SALTA se il fetch iniziale è fallito (`setsLoaded`), o il delete+reinsert cancellerebbe set mai visti.
- **Metriche percepite** (v30): colonne `rpe` (1-10) e `mood` (1-5) su `activities`, nullable per sempre — nessun default che finga un valore mai inserito. Logica in `lib/perceivedMetrics.ts` (`rpeZone` riusa `getZoneByPercent`: l'RPE è la controparte soggettiva dell'intensità da MET; `MOOD_OPTIONS` con icone lucide `Frown`→`Laugh`). UI condivisa `PerceivedMetricsFields` (Log + ActivityEditModal): componente CONTROLLATO dall'esterno, non registrato su react-hook-form, perché i campi devono poter tornare a `null`. Lo slider usa `.spectrum-slider` (la traccia È lo spettro Zone).
- **Recupero: riposo, idratazione, sonno** (v33): tabella `recovery_logs`, una riga per utente per giorno (upsert su `user_id,day`) — `rest` boolean, `water_ml` e `sleep_hours` nullable per sempre. Il giorno di riposo protegge la streak COME un freeze ma è gratuito e si dichiara SOLO per il giorno corrente — retrodatarlo renderebbe inutili i freeze a 300 crediti; limite 2 a settimana lato client (`canMarkRest` in `lib/recovery.ts`). `useRecovery`: fetch 180 giorni, upsert ottimistico con rollback, espone `restDates`. UI `RecoveryCard` in Home (colori acqua `#38BDF8` e notte `#A78BFA` semantici fissi come la scala BMI); Calendar mostra pallino viola + badge; l'offerta freeze NON compare se ieri era riposo; il cron serale salta chi ha `rest=true` oggi (`getRomeToday`). Le medaglie restano su sole attività (i freeze già non contavano).
- **Piani e programmi** (v34): i programmi sono TEMPLATE nel codice (`PLAN_CATALOG` in `lib/plans.ts`); il DB registra solo l'iscrizione (`plan_enrollments`, UNA attiva per utente via indice parziale). Avanzamento MAI persistito: `computePlanProgress` lo deriva dalle attività — settimane a blocchi di 7 giorni da `started_on`, una sessione si sblocca nella SUA settimana (niente front-load), recupero arretrati ammesso, ogni attività salda al massimo una sessione. Crediti col trigger `award_plan_credits` (bounded 0..300, una tantum, non annullabile); `started_on` ancorato a oggi ±1 nella policy INSERT (niente completamenti retroattivi). Client: `useTrainingPlan`, pagina `/plans`, widget in Home "Il tuo percorso" (unico entry point, niente Navbar).
- **Obiettivi personali flessibili** (v36): tabella `personal_goals` — metrica (`sessions`/`minutes`/`km`/`kcal`) + `target` + `activity_type` opzionale + finestra `starts_on`/`ends_on`. Avanzamento derivato da `computeGoalProgress` (`lib/goals.ts`). NIENTE crediti (un obiettivo auto-imposto sarebbe truccabile) e niente policy UPDATE (delete+ricrea). Max 5 attivi lato client; per la metrica km il picker sport filtra su `hasDist` (un obiettivo che non può avanzare non deve nascere). Client: `usePersonalGoals`, widget `GoalsCard` in Home, creazione in `GoalCreateModal` (preset settimana/mese/personalizzato).

### Social e notifiche

- **Social/moderazione** (v26): commenti (`activity_comments`), blocchi (`user_blocks` + funzione security definer `is_blocked_between()` innestata nelle policy INSERT di messages/friendships), segnalazioni insert-only (`user_reports`), classifica globale via RPC `get_global_leaderboard` (solo aggregati settimanali, mai attività grezze). Client: `useComments`, `useBlocks`, `useLeaderboard(scope)`. **Io vs te** (v44): confronto della settimana in corso (lun→oggi, Europe/Rome) nel profilo amico via RPC `get_weekly_comparison` — modello `get_public_profile_stats` ma più restrittivo, risponde SOLO tra amici accettati (il dettaglio settimanale non è per estranei); card "tiro alla fune" in FriendProfileView, nascosta pre-migrazione o senza amicizia.
- **Reazioni sul feed** (v31): il like binario di `activity_likes` (v11) è diventato reazione tipizzata — colonna `kind` a 5 valori (`heart`/`muscle`/`fire`/`clap`/`rocket`, default `heart`), UNA per utente per attività: cambiarla = upsert su `activity_id,user_id`. Logica in `lib/reactions.ts` (aggregazione, optimistic update con rollback, `topKinds`); persistenza in `useFeed.react()`, tollerante pre-migrazione (colonna assente → tutto vale ❤️). Le reazioni sono deliberatamente emoji (linguaggio sociale, come livelli/medaglie — NON iconografia attività). Niente push: aspettano il centro notifiche.
- **Push**: insert su `messages`/`friendships` → webhook Supabase (header `x-webhook-secret`) → `api/webhooks/*` → web-push. Bloccare l'insert (rate limit) blocca anche la notifica. Promemoria serale in `api/cron/reminder.ts` (cron Vercel 20:00 UTC). Cron e webhook rispettano le preferenze granulari (v28): 3 toggle per categoria (`notif_*_enabled`) + fascia di silenzio comune (`notif_quiet_start/end`, fuso Europe/Rome, avvolge la mezzanotte). Logica pura in `api/_lib/notificationPrefs.ts` + `api/_lib/time.ts`; invio effettivo via `sendPushToUserIfAllowed` (`api/_lib/push.ts`). UI in `NotificationSettingsCard.tsx`.
- **Rate limiting**: trigger Postgres `enforce_insert_rate_limit()` (v23+v24) su messages/group_messages/friendships, conteggio sul registro insert-only `rate_limit_events`. L'errore ha prefisso `RATE_LIMIT`, riconosciuto da `lib/errors.ts` → toast dedicati. Sulle `/api` c'è anche un limiter per IP best-effort (`api/_lib/rateLimit.ts`).
- **Eventi stagionali** (v39): classifica a tempo aperta a TUTTA la community (a differenza dei duelli v37, solo amici/gruppo). Gli eventi (finestra di date, metrica, sport opzionale) sono un catalogo nel codice (`SEASONAL_EVENTS` in `lib/seasonalEvents.ts`, stesso pattern di `PLAN_CATALOG` — nessuna ricorrenza automatica per anno, si aggiunge una voce per lanciare il prossimo; catalogo attuale fino a `inverno-2026`, chiude il 31/01/2027 — poi servono voci nuove). ATTENZIONE: `SEASONAL_WINDOWS` in `api/_lib/seasonalPodium.ts` è il GEMELLO server (solo chiavi+finestre, per la notifica podio dal cron): va aggiornato in coppia, il test `seasonalPodium.test.ts` fallisce se divergono. Il DB (`seasonal_claims`) registra solo il riscatto one-shot del podio a finestra chiusa: rank e crediti sono CALCOLATI dal trigger `guard_seasonal_claim` dai dati reali di `activities`, mai fidati dal client (stesso modello fiduciario di `guard_duel_update`). Classifica live via RPC `get_seasonal_leaderboard`. UI in fondo alla pagina Sfide, sotto `DuelsSection`.
- **Centro notifiche in-app** (v40, DELETE in v41, duelli/podi in v45): cronologia persistente per gli eventi che push non copre o copre solo effimeramente — richiesta di amicizia, amicizia accettata, reazione ricevuta, commento ricevuto, level-up; dalla v45 anche duelli (`duel_invite`/`duel_accepted`/`duel_finished`, trigger su `duels` — il rifiuto resta deliberatamente silenzioso, chi chiude non viene rinotificato) e `seasonal_podium` (scritto dal CRON serale con service role, non da un trigger: nulla scatta "allo scadere" di una finestra; idempotente via check su payload->event_key, rank ricalcolato con gli stessi criteri di `guard_seasonal_claim`). Tabella `notifications` scritta SOLO da trigger `security definer` sulle tabelle sorgente (`friendships`, `activity_likes`, `activity_comments`, `duels`, e dentro la RPC `unlock_next_level`) o dal service role: il client non ha una policy INSERT, può leggere le proprie righe, segnarle lette (trigger `guard_notification_update` blocca ogni altra modifica) ed eliminarle (policy DELETE v41 — a differenza dell'update, cancellare la propria cronologia non protegge nulla). Reazioni/commenti rispettano `is_blocked_between`. Client: `context/NotificationsContext.tsx` (badge non lette + realtime via `postgres_changes`, stesso meccanismo di `UnreadContext`), `components/NotificationBell.tsx` — un modale centrato (`.modal-pop`, NON un dropdown ancorato: la campanella è al bordo estremo della TopBar, un dropdown lì si sarebbe steso su quasi tutto lo schermo) che segna tutto letto all'apertura, elimina una singola voce senza conferma (basso rischio) o tutte con doppio tocco (pattern `GoalsCard`). Instradamento del tap in `lib/notifications.ts` (`notificationTarget`, testato) — dal pilastro 04 della roadmap v3 è un DEEP-LINK vero: prende la notifica intera (non solo il tipo) e porta all'attività esatta nel feed (scroll + outline acceso via state di rotta `activityId`) o alla sezione giusta di Sfide (`section: duels/seasonal`, ancore `#duels-section`/`#seasonal-section`). Anche le push deep-linkano: `?dm=<senderId>` apre la conversazione esatta e `?tab=friends` la scheda Amici (query param, non state: il service worker naviga con un vero page load; Social li consuma e pulisce l'URL con replaceState).

### Piattaforma

- **GDPR**: pagine pubbliche `/privacy` e `/termini` (fuori da ProtectedRoute); export dati client-side (`lib/dataExport.ts`); cancellazione account in `api/account/delete.ts` (identità dal JWT, mai dal body); `profiles.terms_accepted_at` NULL = ConsentGate bloccante.
- **Guida in-app** (`/guide`, `pages/Guide.tsx`): la "wiki" di tutte le funzionalità, contenuto statico nel namespace i18n `guide` (sezioni `<details>`/`<summary>` nativi). Raggiungibile dalla card GUIDA nel Profilo e dal link nell'annuncio Novità. **Quando si aggiunge una funzionalità visibile all'utente, aggiornare la sezione corrispondente della guida (o crearne una nuova) fa parte del lavoro.**
- **Performance & bundle** (roadmap v2, pilastro 05): `vite.config.ts` isola `@supabase/supabase-js` e `recharts` in vendor chunk dedicati via `manualChunks` — senza, Rollup/Rolldown li duplicava per intero dentro OGNI chunk che li importava (es. `dataExport` e `useProfile` pesavano rispettivamente 361kB e 233kB solo per questo, ridotti a 3kB e 32kB). Se una libreria pesante nuova finisce importata da più punti, valutare se merita lo stesso trattamento. `npm run build:analyze` genera `stats.html` (treemap gzip/brotli, mai in `dist/`, gitignored) per verificare la composizione dei chunk prima di aggiungere dipendenze. Sentry (`lib/sentry.ts`) ha anche `browserTracingIntegration` con `tracesSampleRate: 0.2`: cattura i Web Vitals (LCP/CLS/INP/TTFB) sulla transazione di pageload, sample rate basso apposta per non consumare la quota free di sentry.io con le transazioni (gli errori restano sample rate 100%, quelli non si limitano).

## Convenzioni in `api/`

- Import relativi **con estensione `.js`** (`from '../_lib/sentry.js'`) — ESM su Vercel, dimenticarla causa ERR_MODULE_NOT_FOUND in produzione (già successo).
- Ogni handler è avvolto in `withSentry` (`api/_lib/sentry.ts`); errori non-fatali via `captureError`.
- `api/_lib/push.ts` esporta `supabaseAdmin` ma al load inizializza web-push (richiede le VAPID env): nelle funzioni che non inviano push, creare un client admin locale invece di importarlo.
- Env server: `SUPABASE_SERVICE_ROLE_KEY`, `SUPABASE_WEBHOOK_SECRET`, `VAPID_*`, `SENTRY_*`; l'URL Supabase si legge da `VITE_SUPABASE_URL` (condivisa col client).

## Gotcha noti

- Le env var inlined nel `define` di `vite.config.ts` hanno `.trim()` apposta: una newline finale nella key su Vercel rompeva SOLO il WebSocket realtime (sintomo: `%0A` nell'URL in console).
- `tsconfig.json` ha `include: ["src"]` — errori in `api/` non emergono dal type-check standard. E `typecheck:api` elenca i file **a mano** in `package.json`: una nuova funzione in `api/` va aggiunta allo script, o non verrà mai type-checkata.
- Il service worker (`public/sw.js`) è cache-first: dopo un deploy l'HTML vecchio può sopravvivere finché la PWA non viene riaperta da zero.
- RLS su `profiles` permette la SELECT a tutti gli utenti autenticati (serve per la ricerca amici): non mettere dati sensibili in quella tabella senza rivedere le policy.
- PowerShell 5.1 spezza gli argomenti con virgolette doppie interne passati a git: per i messaggi di commit multilinea usare `git commit -F <file>`.
- `src/main.tsx` importa `./lib/sentry` come PRIMA riga apposta (registra i global handler prima di ogni altro codice): riordinare gli import lì rompe il monitoraggio errori in produzione senza sintomi visibili in locale (Sentry è no-op senza `VITE_SENTRY_DSN`, quindi il problema non si nota finché non manca un errore reale in produzione).
- Vite 8 usa Rolldown (non Rollup) come bundler: `build.rollupOptions.output.manualChunks` **deve** essere una funzione `(id) => ...`, non l'oggetto `{ nomeChunk: ['pacchetto'] }` della doc classica di Rollup — quella forma fallisce la build con `TypeError: manualChunks is not a function`.
