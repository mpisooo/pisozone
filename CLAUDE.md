This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Cos'ГЁ

PisoZone: PWA in italiano per il tracciamento dell'attivitГ  fisica, con social (amici, messaggi, gruppi, feed), gamification (crediti, 10 livelli, 18 medaglie, sfide giornaliere, streak con "freeze") e notifiche push. Tutta l'UI e i messaggi di commit sono in italiano.

## Comandi

```bash
npm run dev                 # dev server Vite (http://localhost:5173)
npm run build               # tsc + build di produzione
npm test                    # Vitest (vitest.config.ts dedicata; un file: npx vitest run src/lib/challenges.test.ts)
npm run typecheck           # type-check del client (copre SOLO src/)
npm run typecheck:api       # type-check delle funzioni /api in NodeNext (intercetta import senza .js)
```

CI in `.github/workflows/ci.yml`: typecheck Г—2 + test + build a ogni push/PR вЂ” ГЁ il semaforo da controllare prima del deploy (che resta manuale). Niente lint configurato. `gh` CLI non installata: stato delle run via `curl https://api.github.com/repos/mpisooo/pisozone/actions/runs?per_page=1`.

## Deploy вЂ” ATTENZIONE

Il progetto Vercel `pisozone-app` **NON ГЁ collegato a GitHub**: il push su main non deploya nulla. Deploy manuale:

```bash
npx vercel --prod --yes     # CLI giГ  autenticata; produzione = https://pisozone-app.vercel.app
```

Le migrazioni DB (`supabase-schema/supabase-schema-vN.sql`, incrementali, numerate) **le esegue l'utente a mano nel SQL editor di Supabase** вЂ” Claude non ha accesso SQL diretto. Ogni modifica allo schema = nuovo file vN+1, mai modificare i file giГ  eseguiti. Scrivere codice client tollerante all'ordine deploy/migrazione (es. campi opzionali nel tipo finchГ© la colonna non esiste).

## Architettura

**Client React 19 + Vite + Tailwind v4 (no config file) che parla direttamente a Supabase via RLS.** Non c'ГЁ un backend proprio: la logica server vive in (a) trigger e RPC Postgres, (b) poche funzioni serverless Vercel in `api/` con service role.

- **Auth**: username+password su Supabase Auth con email finta `{username}@pisozone.local`. L'email di recupero, se verificata, SOSTITUISCE quella finta in `auth.users.email`.
- **Stato condiviso**: `context/ProfileContext.tsx` ГЁ l'unica fonte del profilo (`hooks/useProfile.ts` ГЁ solo un re-export). Il suo `updateProfile` fa **upsert dell'oggetto completo** con default espliciti: aggiungere un campo a `Profile` richiede attenzione (campo opzionale se la colonna puГІ non esistere ancora).
- **Streak**: UNICA implementazione in `lib/challenges.ts` в†’ `calcStreak(activities, frozenDates)` con i freeze da `useStreakFreeze`. Non ricalcolarlo a mano (regressione giГ  successa in Calendar.tsx).
- **Sfide giornaliere**: `hooks/useDailyChallenges.ts` ГЁ l'unica fonte (Home e Challenges la condividono); i template in `lib/challenges.ts` (`CHALLENGE_POOL`, selezione deterministica per utente+data via PRNG). Le sfide devono essere completabili in giornata (mai basate su streak multi-giorno).
- **Crediti/livelli/temi/cornici**: mutazioni SOLO via RPC atomiche Postgres (`unlock_next_level`, `purchase_theme`, ...), mai update diretti dei crediti.
- **Push**: insert su `messages`/`friendships` в†’ webhook Supabase (header `x-webhook-secret`) в†’ `api/webhooks/*` в†’ web-push. Bloccare l'insert (rate limit) blocca anche la notifica. Il promemoria serale (`api/cron/reminder.ts`, cron Vercel alle 20:00 UTC) e i due webhook rispettano le preferenze granulari dell'utente (v28): 3 toggle per categoria (`notif_reminder_enabled`/`notif_messages_enabled`/`notif_friend_requests_enabled`) + una fascia oraria di silenzio comune (`notif_quiet_start`/`notif_quiet_end`, fuso Europe/Rome, avvolge la mezzanotte). Logica pura e testata in `api/_lib/notificationPrefs.ts` (`allowsNotification`/`isQuietHour`) + `api/_lib/time.ts` (`getRomeHour`/`getRomeTodayRange`, condivisi da cron e webhook); l'invio effettivo passa da `sendPushToUserIfAllowed` in `api/_lib/push.ts`. UI in `NotificationSettingsCard.tsx`.
- **Rate limiting**: trigger Postgres `enforce_insert_rate_limit()` (v23+v24) su messages/group_messages/friendships, conteggio sul registro insert-only `rate_limit_events`. L'errore ha prefisso `RATE_LIMIT`, riconosciuto da `lib/errors.ts` в†’ i hook mostrano toast dedicati. Sulle `/api` c'ГЁ anche un limiter per IP best-effort (`api/_lib/rateLimit.ts`).
- **GDPR**: pagine pubbliche `/privacy` e `/termini` (fuori da ProtectedRoute); export dati client-side (`lib/dataExport.ts`); cancellazione account in `api/account/delete.ts` (identitГ  dal JWT, mai dal body); `profiles.terms_accepted_at` NULL = ConsentGate bloccante.
- **Tracciamento GPS** (v29, roadmap punto 12): "a schermo acceso" per corsa/bici/camminata (`GPS_TRACKABLE_TYPES`) вЂ” il background reale non ГЁ fattibile su iOS PWA standalone (WebKit sospende la pagina a schermo bloccato), quindi serve tenere lo schermo attivo. `useGpsTracking` orchestra `watchPosition` + Wake Lock, con tempo trascorso calcolato sempre da timestamp reali (mai da un contatore) cosГ¬ una sospensione imprevista non lo sfasa. Se un namespace esistente non si adatta bene, crearne uno nuovo seguendo lo stesso pattern (oggetto tipizzato, `export default`, import diretto) e aggiungerlo a `index.ts`.

## Convenzioni in `api/`

- Import relativi **con estensione `.js`** (`from '../_lib/sentry.js'`) вЂ” ESM su Vercel, dimenticarla causa ERR_MODULE_NOT_FOUND in produzione (giГ  successo).
- Ogni handler ГЁ avvolto in `withSentry` (`api/_lib/sentry.ts`); errori non-fatali via `captureError`.
- `api/_lib/push.ts` esporta `supabaseAdmin` ma al load inizializza web-push (richiede le VAPID env): nelle funzioni che non inviano push, creare un client admin locale invece di importarlo.
- Env server: `SUPABASE_SERVICE_ROLE_KEY`, `SUPABASE_WEBHOOK_SECRET`, `VAPID_*`, `SENTRY_*`; l'URL Supabase si legge da `VITE_SUPABASE_URL` (condivisa col client).

- Design system in `src/index.css`: classi `.card`, `.btn-primary`, `.input-dark`, `.skeleton`, heatmap `heatmap-0..4`. Font: Bebas Neue per i titoli (`.font-bebas`), Inter per il body. Gli stili inline `style={{...}}` sono la convenzione del codebase (ignorare i warning del linter).
- Materiali e profonditГ : `.card-hero` (mesh + grana, vedi Sistema Zone sopra) e `--glow-accent` (glow direzionale permanente su `.btn-primary`/`.btn-save`, non solo su `:hover` вЂ” su mobile l'hover non esiste) vanno usati con parsimonia sui momenti di punta, non come default.
- Movimento e aptica (roadmap v2, pilastro 01 punto 4): feedback tattile SOLO via `haptic('light'|'success'|'celebrate'|'error')` da `lib/haptics.ts`, MAI `navigator.vibrate` diretto вЂ” su iOS (il dispositivo dell'utente) non esiste e il modulo ripiega sul tick nativo dell'`<input type="checkbox" switch>` di Safari 17.4+. Token di movimento in `index.css` (`--dur-fast/base/slow`, `--ease-out`, `--ease-spring`): i modali entrano con `.overlay-fade` sul backdrop + `.modal-pop` (dialoghi centrati) o `.sheet-up` (bottom sheet); le schermate full-screen restano su `.page-enter`. I numeri importanti contano con `components/AnimatedNumber.tsx` (rAF, niente dipendenze). Navigazione con `viewTransition` (react-router) su Navbar e card di Home, chrome fermo via `view-transition-name` su `.topbar`/`.navbar` вЂ” progressive enhancement, dove non supportata resta `.page-enter`. `prefers-reduced-motion` ГЁ rispettato globalmente in coda a `index.css` (azzera anche le transizioni inline e le view transitions; AnimatedNumber fa lo stesso check via matchMedia): le animazioni nuove non devono aggiungere guardie proprie.
- Stati di caricamento SOLO con `SkeletonCard` / `SkeletonRow` (entrambi in `components/SkeletonCard.tsx`) вЂ” mai `animate-pulse` con div ad hoc (ripulito col punto 15).
- Ogni modale/popup DEVE seguire il pattern accessibile: `useFocusTrap(ref, open, onClose)` + `role="dialog"` + `aria-modal` + chiusura con Esc (riferimento: `ActivityEditModal.tsx`). Gli overlay auto-dismissivi usano invece `role="status"` + `aria-live`.
- Overlay su mobile (l'utente usa la PWA standalone su iPhone, `viewport-fit=cover`): elementi interattivi vicino ai bordi SEMPRE dentro la safe area (`env(safe-area-inset-top/bottom)` вЂ” la X di un lightbox a `top-4` finisce sotto orologio/batteria). Form lunghi = schermata fissa a tutto schermo con header sticky (mai bottom sheet che scorre tutto insieme). Ogni overlay con scroller interno DEVE bloccare il body (`document.body.style.overflow='hidden'` in useEffect con ripristino) e usare `overscroll-contain` + `min-h-0` sullo scroller, altrimenti su iOS lo scroll chaining aggancia la pagina dietro e il gesto "si blocca" a metГ . Riferimento per tutti e tre: `ActivityEditModal.tsx`.
- Rotte in `App.tsx`: pubbliche (`/auth`, `/privacy`, `/termini`) fuori da ProtectedRoute; il resto dentro AppLayout con lazy loading per pagina.
- Errori utente: messaggi in italiano, concreti, con invito a riprovare (mai `error.message` grezzo).

## Gotcha noti

- Le env var inlined nel `define` di `vite.config.ts` hanno `.trim()` apposta: una newline finale nella key su Vercel rompeva SOLO il WebSocket realtime (sintomo: `%0A` nell'URL in console).
- `tsconfig.json` ha `include: ["src"]` вЂ” errori in `api/` non emergono dal type-check standard.
- Il service worker (`public/sw.js`) ГЁ cache-first: dopo un deploy l'HTML vecchio puГІ sopravvivere finchГ© la PWA non viene riaperta da zero.
- RLS su `profiles` permette la SELECT a tutti gli utenti autenticati (serve per la ricerca amici): non mettere dati sensibili in quella tabella senza rivedere le policy.
