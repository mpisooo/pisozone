# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Cos'è

PisoZone: PWA per il tracciamento dell'attività fisica — 20 sport, GPS, log palestra con record personali, programmi multi-settimana, obiettivi liberi, recupero (riposo/acqua/sonno) — con social (amici, messaggi, gruppi, feed con reazioni), gamification (crediti, 10 livelli, 23 medaglie, sfide giornaliere, streak con "freeze") e notifiche push. UI disponibile in italiano (default) e inglese (switch in Profilo, v3 pilastro 04); i messaggi di commit restano sempre in italiano.

## Comandi

```bash
npm run dev                 # dev server Vite (http://localhost:5173)
npm run build               # tsc + build di produzione
npm test                    # Vitest (vitest.config.ts dedicata; un file: npx vitest run src/lib/challenges.test.ts)
npm run typecheck           # type-check del client (copre SOLO src/)
npm run typecheck:api       # type-check delle funzioni /api in NodeNext (intercetta import senza .js)
npm run build:analyze       # build + stats.html (treemap dei chunk, mai committato/servito)
npm run test:e2e            # Playwright (solo chromium), smoke test in e2e/ — vedi e2e/README.md per lo scope
```

CI in `.github/workflows/ci.yml`: typecheck ×2 + test + build + test:e2e a ogni push/PR — è il semaforo da controllare prima del deploy (che resta manuale). Niente lint configurato. `gh` CLI non installata: stato delle run via `curl https://api.github.com/repos/mpisooo/pisozone/actions/runs?per_page=1`.

## Deploy — ATTENZIONE

Il progetto Vercel `pisozone-app` **NON è collegato a GitHub**: il push su main non deploya nulla. Deploy manuale:

```bash
npx vercel --prod --yes     # CLI già autenticata; produzione = https://pisozone-app.vercel.app
```

Le migrazioni DB (`supabase-schema/supabase-schema-vN.sql`, incrementali, numerate) **le esegue l'utente a mano nel SQL editor di Supabase** — Claude non ha accesso SQL diretto. Ogni modifica allo schema = nuovo file vN+1 (guardare la cartella per l'ultimo numero), mai modificare i file già eseguiti. Scrivere codice client tollerante all'ordine deploy/migrazione (es. campi opzionali nel tipo finché la colonna non esiste; fetch fallito su tabella nuova = dati vuoti, mai crash).

## Architettura

**Client React 19 + Vite + Tailwind v4 (no config file) che parla direttamente a Supabase via RLS.** Non c'è un backend proprio: la logica server vive in (a) trigger e RPC Postgres, (b) poche funzioni serverless Vercel in `api/` con service role.

### Fondamenta

- **Auth**: username+password su Supabase Auth con email finta `{username}@pisozone.local`. L'email di recupero, se verificata, SOSTITUISCE quella finta in `auth.users.email`.
- **Stato condiviso**: `context/ProfileContext.tsx` è l'unica fonte del profilo (`hooks/useProfile.ts` è solo un re-export). Il suo `updateProfile` fa **upsert dell'oggetto completo** con default espliciti: aggiungere un campo a `Profile` richiede attenzione (campo opzionale se la colonna può non esistere ancora; non includerlo nell'update finché `profile.<campo> === undefined`, o l'upsert fallirebbe pre-migrazione).
- **Logica in moduli puri**: i calcoli vivono in `src/lib/` testati con Vitest (`challenges`, `stats`, `zones`, `gps`, `exerciseSets`, `plans`, `goals`, `recovery`, `reactions`, ...), le pagine restano sottili. Le funzioni con dipendenza dal tempo prendono `now` come parametro per test deterministici. `lib/stats.ts` genera anche il CSV per Excel italiano (`;` e virgola decimale); `lib/dataExport.ts` ha i download JSON/CSV (BOM UTF-8 per Excel) e l'export GDPR completo (le tabelle post-v31 stanno in `optionalTables`: se mancano pre-migrazione l'export non fallisce; `activity_routes` è PAGINATA a blocchi da 1000 — PostgREST tronca silenziosamente oltre quel tetto e i percorsi GPS sono l'unica tabella che lo sfonda).
- **Streak**: UNICA implementazione in `lib/challenges.ts` → `calcStreak(activities, protectedDates)` dove protectedDates = freeze da `useStreakFreeze` + giorni di riposo da `useRecovery` (v33): tutti e tre i chiamanti (Home, Calendar, Challenges) passano `[...frozenDates, ...restDates]`. Non ricalcolarlo a mano (regressione già successa in Calendar.tsx).
- **Coda offline** (roadmap v2, pilastro 05, "offline-first robusto"): `addActivity` in `hooks/useActivities.ts` mette in coda (localStorage) un'attività se `!navigator.onLine` o se l'insert fallisce per un vero errore di rete (`postgrest-js` risponde con `status: 0` SOLO in quel caso — mai per un vero codice HTTP: vedi `isNetworkFailure` in `lib/offlineQueue.ts`). NIENTE Background Sync nel service worker: su iOS Safari (il dispositivo dell'utente) l'API non esiste — stesso limite già noto per il GPS "a schermo acceso". Si riprova quando l'app è davvero aperta (evento `online` + `visibilitychange`), non in background reale. Le attività in coda appaiono subito in UI come attività "pending" (`mergeWithPending`, id sintetico `pending-<uuid>`, crediti a zero finché non sincronizzate) — TUTTE le pagine che usano `useActivities()` (Home, Calendar, Stats, Challenges, Plans, Medals) le vedono già incluse nei loro calcoli, comportamento voluto (ottimistico). Sono modificabili ed eliminabili dal Calendario come le altre (roadmap v3, pilastro 04, "offline senza asterischi"): `updateActivity` in `useActivities` rileva l'id sintetico e aggiorna il payload dentro la coda invece di chiamare Supabase; il badge "In attesa di rete" resta, ma il tocco non è più disabilitato. Foto ed esercizi allegati mentre offline non vengono più scartati: `lib/offlineAttachments.ts` li tiene in IndexedDB (un `Blob` non è serializzabile nel JSON di `offlineQueue`/localStorage), indicizzati per `localId`; `flushQueue` li applica best effort appena l'attività ottiene un id vero (stesso pattern "dopo l'insert" di foto/esercizi/percorsi online), poi ripulisce la voce IndexedDB — anche se l'attività viene scartata per un errore vero, per non lasciare allegati orfani.
- **Sfide giornaliere**: `hooks/useDailyChallenges.ts` è l'unica fonte (Home e Challenges la condividono); i template in `lib/challenges.ts` (`CHALLENGE_POOL`, selezione deterministica per utente+data via PRNG). Le sfide devono essere completabili in giornata (mai basate su streak multi-giorno).
- **Crediti/livelli/temi/cornici**: mutazioni SOLO via RPC atomiche Postgres (`unlock_next_level`, `purchase_theme`, `use_streak_freeze`, ...) o trigger `security definer` (`award_challenge_credits`, `award_plan_credits`), mai update diretti dei crediti dal client.
- **UI one-shot via flag su `profiles`**: `push_prompt_seen` (prompt push), `terms_accepted_at` NULL → ConsentGate, `onboarding_seen` false → OnboardingTour, `news_seen_version` (v35) < `NEWS_VERSION` → `WhatsNewOverlay` (annuncio novità per utenti esistenti: VERSIONATO — per una nuova ondata si incrementa la costante in `WhatsNewOverlay.tsx` e si riscrivono gli item in `shell.whatsNew`; il tour al completamento allinea anche news_seen_version, ma SOLO se la colonna esiste). Stesso pattern per flag futuri: colonna con default, campo opzionale nel tipo TS finché la migrazione non è eseguita.
- **i18n + switch lingua IT/EN** (v3, pilastro 04, chiuso il 20/07/2026): tutte le stringhe UI vivono in `src/lib/i18n/` — un file per namespace (`common`, `shell`, `auth`, `home`, `log`, `calendar`, `stats`, `challenges`, `medals`, `profile`, `social`, `legal`, `recovery`, `plans`, `guide`, `goals`, `insights`, `wrapped`, `share`, `duels`, `seasonalEvents`, `notifications`, `recap`, `heatmap`, `segments`, `routines`, `sportPicker`, `landing` — 28 in totale), niente funzione `t('a.b.c')` a stringa. Ogni file ha DUE oggetti, `it` (`as const`) ed `en` (`Widen<typeof it>` — vedi sotto), esportati come `createNamespaceProxy(it, en)`: le pagine continuano a importare il namespace direttamente e staticamente (`import log from '../lib/i18n/log'`, non da un barrel), il Proxy sceglie l'oggetto giusto ad ogni lettura in base a `getLanguage()` — nessuna pagina è mai stata convertita a un hook `useStrings()`. `src/lib/i18n/language.ts` tiene lo stato (`localStorage` chiave `pz-language`, letture avvolte in try/catch perché Vitest gira in ambiente `node` senza `localStorage` e diversi moduli `lib/` puri — `dataExport`, `stats`, `shareCard`, `insights`, `perceivedMetrics` — importano namespace i18n) + `useLanguage()` (`useSyncExternalStore`). Il cambio lingua in tempo reale (switcher in Profilo → `LanguageSettingsCard.tsx`) non aggiorna le pagine già montate da solo: `AppLayout` in `App.tsx` chiama `useLanguage()` e mette `key={language}` sul `<div className="app-layout">`, forzando il remount dell'intero sottoalbero visibile (TopBar/Routes/Navbar) a ogni switch — badge/notifiche restano fuori dalla key, niente riconnessione realtime. **`Widen<T>` in `proxy.ts`**: `it` è `as const`, quindi ogni sua stringa è un tipo literal (`'Salva'`, non `string`) — `en: typeof it` non compila (`'Save'` non è assegnabile a `'Salva'`); `en` va sempre annotato `Widen<typeof it>` (allarga solo le foglie stringa e i tipi di ritorno delle funzioni a `string`, preservando struttura/nesting/arità — le chiavi mancanti o in eccesso restano un errore di compilazione). `it.test.ts` verifica sia che ogni foglia (in entrambe le lingue) sia stringa/funzione non vuota, sia che `en` rispecchi esattamente la forma di `it` namespace per namespace. Gli import di `date-fns/locale` restano invariati (formattazione date ≠ stringhe UI: le date sono in italiano anche con l'app in inglese, limite noto e accettato) e i numeri usano `toLocaleString('it-IT'|'en-US', ...)` a seconda della lingua dove serve un separatore decimale corretto. **Regola permanente**: ogni stringa nuova va nel namespace corrispondente (nuovo namespace per pagine nuove), SEMPRE in entrambe le lingue — mai testo italiano hardcoded nel JSX, nemmeno "temporaneamente". Eccezione voluta e mai cambiata: i DATI di dominio (titoli di sfide/medaglie/programmi/eventi stagionali in `CHALLENGE_POOL`/`MEDALS`/`PLAN_CATALOG`/`seasonalCatalog.ts`) restano hardcoded in italiano nei loro moduli `lib/`, fuori dal sistema i18n — anche con l'app in inglese quei nomi non si traducono.

Nei badge di sola lettura l'icona mostra sempre il proprio colore (niente più wrapper `text-[var(--red)]`); gli sfondi circolari che prima erano tinti di rosso (Home, FeedTab) sono diventati grigio neutro (`var(--grey-light)`) per non stonare con un'icona di un altro colore. Le barre di riepilogo per sport (Profile/FriendProfileView, "I tuoi numeri per sport") leggono `ACTIVITY_ICON_COLORS` invece di un colore fisso. **Iconografia medaglie** (aggiunta il 22/07/2026): le 23 medaglie hanno invece pittogrammi dedicati, stesso stile "pieno" delle icone sport ma con la classica cornice a medaglia (nastro + disco). A differenza di `ActivityIcon`, qui i colori sono FISSI dentro l'SVG: il nastro/disco segue il TIER (bronzo/argento/oro/diamante, `lib/medalIconPaths.tsx` → `TIER_PALETTES`), il simbolo interno ha un colore proprio per famiglia di missione (corsa blu, palestra viola, streak arancione, montagna verde scuro, ecc.), su una placca bianca che garantisce contrasto qualunque sia il tier. `components/MedalIcon.tsx` prende `medalKey` (il `MedalDefinition.key`, non l'emoji): sostituisce l'emoji in Medals.tsx, Home.tsx (medaglia più vicina), Profile.tsx/FriendProfileView.tsx (bacheca medaglie, roadmap v7) e MedalCelebrationOverlay.tsx. Il campo `MedalDefinition.icon` (emoji) resta nel dato ma non è più usato per il rendering — nessun altro punto del codice lo consumava.
- **Le tre famiglie di icone e quando usare quale** (regola scritta il 23/07/2026, roadmap "PisoZone Next" P1-06, dopo un audit di tutto il codice): (1) **Lucide outline** (`lucide-react`) per navigazione e azioni funzionali — bottoni, header, chiusura modali, frecce; (2) **pittogrammi pieni colorati** (`ActivityIcon`/`MedalIcon`/`EmptyState`) per le entità di dominio dell'app — sport, medaglie, stati vuoti; (3) **emoji di sistema** SOLO per gamification/social/celebrativo — livelli, sfide, programmi, reazioni, mood, podi stagionali: scelta dichiarata, non un accumulo casuale (vedi bullet sopra). **Eccezioni note e accettate, non da "correggere" per coerenza fine a se stessa**: la card "Record" in `Stats.tsx` usa un piccolo set di emoji decorative per tipo di statistica (⏱️ tempo, 🔥 calorie, 📏 distanza) — un linguaggio locale autosufficiente, cambiarne una sola romperebbe la coerenza interna di quella card; il pallino 😴 di "Giornata di riposo" in `Calendar.tsx` porta un significato specifico (il riposo) che nessun pittogramma esistente rende, non sostituito con un pittogramma generico per non perdere quella sfumatura; il 🏆 di `NotificationBell.tsx` per la notifica di podio stagionale (evento di sistema senza un utente-attore, al posto dell'avatar). **Bug reale corretto in questo giro**: `WrappedOverlay.tsx` usava 🔥 per lo streak migliore, mentre Home.tsx e Calendar.tsx usano l'icona Lucide `Flame` colorata per lo stesso identico concetto — corretto a `Flame` anche lì, la vera incoerenza trovata (non l'eccezione di Stats, che è un concetto diverso: calorie, non streak).

### Componenti e interazione

- **Design tokens formalizzati** (roadmap v2, pilastro 05): tutti i token vivono in UNA sezione in cima a `src/index.css` (4 categorie commentate: colore/temi, zone, elevazione, movimento) invece che sparsi. Colore: gli 8 nomi base (`--black`/`--grey*`/`--red*`/`--color-text`/`--accent-rgb`) hanno doppia fonte voluta — `THEME_CSS_VARS` in `lib/levels.ts` è quella vera, applicata a runtime da `ThemeContext`; il blocco `:root`/`html.light` in `index.css` è SOLO il fallback pre-idratazione e va tenuto manualmente allineato a `THEME_CSS_VARS.dark`/`.light` (nessun meccanismo automatico li sincronizza). Zone: `--zone-1..4` + il gradiente `--zone-spectrum` (usato da `.zone-spectrum`/`.spectrum-slider` e specchiato in JS da `ZONE_SPECTRUM_GRADIENT` in `lib/zones.ts`, che punta al token invece di ripeterlo). Elevazione: `--shadow-sm/md/lg`, `--glow-accent`, `--shadow-chrome-down/up`. Classi del design system: `.card`, `.btn-primary`, `.input-dark`, `.skeleton`, heatmap `heatmap-0..4`. Font: Bebas Neue per i titoli (`.font-bebas`), Inter per il body — la scala tipografica resta quella nativa di Tailwind, nessun token proprio. Gli stili inline `style={{...}}` sono la convenzione del codebase.
- Materiali e profondità: `.card` ha elevazione di base ovunque; `.card-hero` (mesh gradient + grana SVG) e `--glow-accent` (permanente su `.btn-primary`/`.btn-save`, non solo `:hover` — su mobile l'hover non esiste) vanno usati con parsimonia sui momenti di punta di una pagina, non come default.
- Movimento e aptica: feedback tattile SOLO via `haptic('light'|'success'|'celebrate'|'error')` da `lib/haptics.ts`, MAI `navigator.vibrate` diretto — su iOS (il dispositivo dell'utente) non esiste e il modulo ripiega sul tick nativo dello switch di Safari 17.4+. Token di movimento in `index.css` (`--dur-fast/base/slow`, `--ease-out`, `--ease-spring`): modali con `.overlay-fade` + `.modal-pop` (centrati) o `.sheet-up` (bottom sheet); schermate full-screen su `.page-enter`. Numeri importanti con `AnimatedNumber.tsx`. Navigazione con `viewTransition` (react-router) su Navbar e card di Home, chrome fermo via `view-transition-name`. `prefers-reduced-motion` è rispettato globalmente in coda a `index.css`: le animazioni nuove non devono aggiungere guardie proprie.
- Stati di caricamento SOLO con `SkeletonCard` / `SkeletonRow` (`components/SkeletonCard.tsx`) — mai `animate-pulse` con div ad hoc.
- **Stati vuoti SOLO con `EmptyState`** (`components/EmptyState.tsx`, pittogramma + `title` + `hint` opzionale + `cta` opzionale, variante `compact` per stati vuoti dentro card già piene) — mai un paragrafo di testo grezzo o un'emoji ad-hoc in un cerchio fatto a mano. Template di copy (roadmap "PisoZone Next" P1-07, scritto il 23/07/2026 dopo un audit di tutti gli stati vuoti del prodotto): `title` = cosa vedi qui (breve, senza colpevolizzare l'utente — "Nessuna sfida in corso", non "Non hai ancora sfidato nessuno"), `hint` = cosa fare per riempirlo SOLO quando non è già ovvio da un bottone visibile subito sopra/sotto (se un CTA come "+ Nuova sfida" è già nell'header della sezione, l'hint può limitarsi a nominarlo invece di duplicare l'azione in un bottone proprio). Applicato sistematicamente a feed/amici/gruppi/chat/routine/segmenti/heatmap (già in EmptyState da roadmap v3) più, in questo giro, bacheca medaglie di `Profile.tsx`, sfide di `DuelsSection.tsx` e centro notifiche di `NotificationBell.tsx` (prima testo semplice o emoji senza pittogramma). **Eccezione accettata**: il pannello "Giornata di riposo" di `Calendar.tsx` resta testo semplice con l'emoji 😴 — non è uno stato "vuoto" nel senso di questa regola ma un'informazione di stato (nessun allenamento quel giorno, che sia un riposo dichiarato o no), e forzarci un CTA "registra un'attività" per un giorno potenzialmente passato sarebbe fuorviante.
- Ogni modale/popup DEVE seguire il pattern accessibile: `useFocusTrap(ref, open, onClose)` + `role="dialog"` + `aria-modal` + chiusura con Esc (riferimento: `ActivityEditModal.tsx`). Gli overlay auto-dismissivi usano invece `role="status"` + `aria-live`.
- Overlay su mobile (l'utente usa la PWA standalone su iPhone, `viewport-fit=cover`): elementi interattivi vicino ai bordi SEMPRE dentro la safe area (`env(safe-area-inset-top/bottom)`). Form lunghi = schermata fissa a tutto schermo con header sticky (mai bottom sheet che scorre tutto insieme). Ogni overlay con scroller interno DEVE bloccare il body (`document.body.style.overflow='hidden'` in useEffect con ripristino) e usare `overscroll-contain` + `min-h-0` sullo scroller, o su iOS lo scroll chaining aggancia la pagina dietro. Riferimento per tutti e tre: `ActivityEditModal.tsx`.
- Rotte in `App.tsx`: pubbliche (`/auth`, `/privacy`, `/termini`) fuori da ProtectedRoute; il resto dentro AppLayout con lazy loading per pagina. **Landing pubblica su "/"** (roadmap "PisoZone Next", pilastro Landing): non una rotta a parte — `ProtectedRoute.tsx` mostra `pages/Landing.tsx` al posto del redirect a `/auth` SOLO quando il pathname è esattamente `/` e l'utente non è autenticato; ogni altra rotta protetta (`/log`, `/calendar`, ...) continua a rimbalzare su `/auth` com'era prima. Pagina pre-login come Auth/Privacy/Termini (sfondo `#0D0D0D` fisso, testo su scala grigia Tailwind fissa, non `var(--color-text)` — coerente con l'eccezione già dichiarata per queste pagine, vedi Identità visiva). CTA "Inizia gratis" naviga a `/auth?tab=register`: `Auth.tsx` legge il query param per preselezionare la scheda di registrazione. Niente prova sociale o screenshot inventati: la striscia di icone sport nell'hero riusa `ActivityIcon` reale, non immagini finte.
- **Anteprima social** (roadmap "PisoZone Next", pilastro Growth, 22/07/2026): `index.html` ha canonical + set completo Open Graph/Twitter Card, puntati a `public/og-image.png` (2400×1260, generata offline con Playwright — non a mano — screenshottando un HTML standalone con Bebas Neue incorporata come `@font-face` in base64 e 5 icone sport reali colorate, stesso stile della landing; lo script generatore era usa-e-getta e non è stato committato, stesso pattern delle icone attività). ## Gotcha noti

- Le env var inlined nel `define` di `vite.config.ts` hanno `.trim()` apposta: una newline finale nella key su Vercel rompeva SOLO il WebSocket realtime (sintomo: `%0A` nell'URL in console).
- `tsconfig.json` ha `include: ["src"]` — errori in `api/` non emergono dal type-check standard. E `typecheck:api` elenca i file **a mano** in `package.json`: una nuova funzione in `api/` va aggiunta allo script, o non verrà mai type-checkata.
- Il service worker (`public/sw.js`) è cache-first: dopo un deploy l'HTML vecchio può sopravvivere finché la PWA non viene riaperta da zero.
- RLS su `profiles` permette la SELECT a tutti gli utenti autenticati (serve per la ricerca amici): non mettere dati sensibili in quella tabella senza rivedere le policy.
- PowerShell 5.1 spezza gli argomenti con virgolette doppie interne passati a git: per i messaggi di commit multilinea usare `git commit -F <file>`.
- `src/main.tsx` importa `./lib/sentry` come PRIMA riga apposta (registra i global handler prima di ogni altro codice): riordinare gli import lì rompe il monitoraggio errori in produzione senza sintomi visibili in locale (Sentry è no-op senza `VITE_SENTRY_DSN`, quindi il problema non si nota finché non manca un errore reale in produzione).
- Vite 8 usa Rolldown (non Rollup) come bundler: `build.rollupOptions.output.manualChunks` **deve** essere una funzione `(id) => ...`, non l'oggetto `{ nomeChunk: ['pacchetto'] }` della doc classica di Rollup — quella forma fallisce la build con `TypeError: manualChunks is not a function`.
